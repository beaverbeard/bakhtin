# Направления для doc

Пул направлений для пятёрки `/бахтин` при `format=doc` (деловой документ:
КП, бриф, описание услуги, memo, one-pager). Каждое направление — конкретный
жанровый референс, не слово-роль.

Если N < 5 — первые N (отсортированы по применимости к деловым документам).
Если N > 5 — добавить вариации (два разных КП под разные типы клиентов, два
memo с разной фактурой).

**Контекст применения.** Этот пул работает на коммерческих предложениях,
брифах, описаниях услуг для лендинга, внутренних memo, one-pager для
встреч. Голос делового документа — нейтрально-уверенный,
voice-плагин обычно не подключается.

**Длина.** Документ — не «текст», его читают по диагонали. Каждое
направление оптимизировано под сканирование: подзаголовки, иерархия,
буллеты обязательны.

---

## Направление 1: Структурированное КП

**Что это.** Структурированное коммерческое предложение: задача клиента →
подход → план работ → команда → стоимость → критерии успеха. Прямой
тон, без воды и инфобизнес-приёмов.

**Эталон.** Типовое деловое КП агентства или консалтинга. Структура
одинаковая, фактура меняется под клиента.

**Как выглядит:**
- Раздел «Задача» — формулировка проблемы клиента его словами
- Раздел «Подход» — гипотеза и почему именно она предлагается (1-2 абзаца)
- Раздел «План» — 3-6 этапов с дедлайнами и артефактами на выходе
- Раздел «Команда» — кто и за что отвечает (имена, не «эксперты»)
- Раздел «Стоимость» — цифры и из чего складываются
- Раздел «Критерии успеха» — как поймём, что сработало

**Как НЕ выглядит:**
- ❌ «О нас» в первом разделе — клиенту неинтересно, ему интересна его
  задача
- ❌ Стоимость без расшифровки («работа агентства — N млн»)
- ❌ Критерии успеха в духе «качественный контент» — нужны измеримые

## Направление 2: Бюро-формат (понимание задачи)

**Что это.** Документ строится вокруг 6 пунктов Горбунова: полезное
действие → аудитория → фактура → пробелы → ограничения → критерии
готовности. Подход «сначала понимаем, потом предлагаем».

**Эталон.** Брифы и документы Бюро Горбунова, школа Бюро формат
«понимание задачи». В частности — шаблон Бахтина для
`drafts/understanding-{slug}.md` (это и есть Бюро-формат).

**Как выглядит:**
- Раздел «Полезное действие» — что должен сделать/подумать читатель
- Раздел «Аудитория и её состояние» — кто и в каком контексте
- Раздел «Что есть» — фактура, источники, кейсы
- Раздел «Что надо узнать» — пробелы, гипотезы
- Раздел «Ограничения» — формат, дедлайн, табу
- Раздел «Критерии готовности» — когда говорим «всё»

**Как НЕ выглядит:**
- ❌ Перевёрнутая логика «сначала решение, потом понимание»
- ❌ Аудитория описана как «B2B клиенты 25-45» — это таргетинг, не
  состояние читателя
- ❌ Критерии без измеримости («чтобы было понятно»)

## Направление 3: Pyramid principle (McKinsey)

**Что это.** Главный вывод вверху, поддерживающие — ниже, детали — ещё
ниже. Читатель прочитал первый абзац — понял главное. Прочитал заголовки
— понял структуру. Прочитал детали — получил доказательную базу.

**Эталон.** Memo McKinsey и BCG, формат Pyramid Principle от Барбары
Минто. Стратегические one-pager в крупных корпорациях. Управленческие
документы внутреннего пользования.

**Как выглядит:**
- Первый абзац (Executive Summary) — главный вывод + 3 опоры
- Каждая опора — отдельный раздел с подзаголовком
- В каждом разделе: тезис → данные → следствие
- Финальный раздел — следующие шаги (Next Steps), 3-5 конкретных пунктов
- Без введения «о компании» и без заключения «таким образом»

**Как НЕ выглядит:**
- ❌ Главный вывод в финале (это эссе, не memo)
- ❌ Описательная структура «введение → история вопроса → разбор → вывод»
- ❌ Развёрнутая презентация без цифр — pyramid principle требует
  доказательной базы

## Направление 4: Короткое memo (YC/Bezos)

**Что это.** Один лист. Проблема → данные → решение → следующие шаги.
Без рамок, без декоративных элементов. Цель: автор должен думать
до того, как пишет, читатель — после.

**Эталон.** Format Amazon 6-pager и 1-pager (Bezos memo), Y Combinator
memo для batch-партнёров, внутренние RFC в Stripe и Linear. Жёсткая
дисциплина «один документ = одно решение».

**Как выглядит:**
- Заголовок — формулировка вопроса/решения в одной фразе
- Раздел «Проблема» — 1-2 абзаца, что не работает и почему
- Раздел «Данные» — цифры, наблюдения, что мы знаем
- Раздел «Решение» — что предлагается, ключевые выборы
- Раздел «Next Steps» — 3-5 конкретных действий с владельцами и сроками
- Общий объём — 1-2 страницы максимум

**Как НЕ выглядит:**
- ❌ Развёрнутый бэкграунд («компания была основана в...»)
- ❌ Решение без альтернатив, которые рассматривались
- ❌ Next Steps без владельцев и сроков — это пожелания, не действия

## Направление 5: Описание услуги для лендинга

**Что это.** Продающий текст для конкретной услуги: польза для клиента →
как работает → пример успеха → цена/CTA. Sales-driven, но без инфобизнес-
приёмов и пустых обещаний.

**Эталон.** Лендинги Бюро Горбунова, Stripe.com product pages, лендинги
Тинькофф Бизнес и Тинькофф B2B. Конкретика во всём: что получит клиент,
за сколько, как именно.

**Как выглядит:**
- Заход — выгода для клиента одной фразой (не «о нас», а «вы получите»)
- Раздел «Как работает» — 3-4 шага процесса с конкретикой
- Раздел «Что входит» — список с конкретными артефактами
- Раздел «Кейс» — пример успеха с цифрами (короткий)
- Раздел «Цена и условия» — прозрачно, без «звоните узнавать»
- CTA — конкретное действие («Записаться на встречу», не «Узнать больше»)

**Как НЕ выглядит:**
- ❌ Заход через «миссию» или «о нас» — клиенту интересна его выгода
- ❌ Кейс без цифр — это история, а не доказательство
- ❌ «Узнать больше» / «Получить консультацию» как CTA — слишком абстрактно

---

## Как использовать

При `format=doc` Бахтин:
1. Спрашивает N агентов.
2. Берёт первые N направлений (или все 5).
3. Если N > 5 — добавляет вариации (например, два КП под разные типы
   клиентов, два memo с разной фактурой).
4. Каждому передаёт: материал + понимание задачи + outline +
   voice-отпечаток + название и описание направления (этот файл).

При фидбэке «не то направление» — автор называет, что убрать или
заменить, скил перезапускает пятёрку с новым набором.
